Ownership & Moves

How Rust frees your memory for you — and refuses to let you use-after-free.

~15 min Concepts → practice ✓ No install — runs in your browser

This is the idea that makes Rust different from every language you've used. Get this, and borrowing, lifetimes, and the safe Rust↔Python boundary all follow. The official book puts it bluntly:

From the Rust Book

"When you understand ownership, you'll have a solid foundation for understanding the features that make Rust unique."1

You learn concepts first here, then you'll predict what the compiler does and run each snippet to check yourself. Nothing to install — every ▶ Run button opens the code, pre-filled, in the Rust Playground.

1The problem you already know

You wrote C and C++. So you already know the hardest unsolved problem in that world: who frees this memory, and when? The book names the exact three ways it goes wrong:

↩ From your C/C++ days

"If we forget, we'll waste memory. If we do it too early, we'll have an invalid variable. If we do it twice, that's a bug too."1

In C terms: forget free → leak. free too early → dangling pointer / use-after-free. free twice → double-free → heap corruption → CVE.

Other languages answer this with a garbage collector (Python, Java, C#) — a runtime that periodically figures out what's safe to free. That costs runtime work and unpredictable pauses. Rust answers it a third way: the compiler tracks who owns each value and inserts the free for you, at compile time, for free:

From the Rust Book

"None of the features of ownership will slow down your program while it's running."1

2Stack and heap — a 30-second refresh

Ownership is really a set of rules about who frees the heap allocation, so let's reload the two memory regions. Nothing here is new to you — it's the same model as C.

StackHeap
Fixed, known size at compile time. LIFO — "a stack of plates."1 Dynamic size. You ask the allocator for space; it finds a spot and hands back a pointer.1
Push/pop is instant — the slot is always "the top." Allocating searches for space; reading means following a pointer — both slower.
i32, bool, char, f64, fixed arrays… String, Vec<T>, Box<T> — anything growable.

Key picture: a Rust String is a little 3-field struct on the stack — a pointer to the bytes, a length, and a capacity — pointing at the actual text on the heap. Hold that image; the whole lesson turns on it.

3The three rules of ownership

This is the entire model. Verbatim from the book1:

  1. Each value in Rust has an owner.
  2. There can be only one owner at a time.
  3. When the owner goes out of scope, the value is dropped (freed).

"Goes out of scope" means execution reaches the closing } the value was declared in. At that point Rust automatically calls a function named drop — the value's cleanup code — for you. You never write free.

↩ This is RAII — the book says so

"In C++, this pattern of deallocating resources at the end of an item's lifetime is sometimes called Resource Acquisition Is Initialization (RAII). The drop function in Rust will be familiar to you if you've used RAII patterns."1

So: it's the destructor-at-end-of-scope you knew from C++ — but now it's the language-wide default, not something you opt into with a smart-pointer class.

fn main() {
    {
        let s = String::from("hello"); // s owns heap text from here
        // ... use s ...
    } // <- s goes out of scope: drop(s) runs, heap freed. Automatically.
}

It compiles and does nothing visible — that's the point. The free happened, and you didn't write it.

4The surprise: assignment can move

Here's where Rust diverges from everything you know. Watch what happens when you assign one String to another:

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;

    println!("{s1}, world!"); // will this compile?
}

In C++, s2 = s1 would copy the struct's three fields, leaving two pointers to the same heap text — and at scope end both destructors would free it. That's a double-free. The book is explicit about the danger:

From the Rust Book

"When s2 and s1 go out of scope, they will both try to free the same memory. This is known as a double free error… Freeing memory twice can lead to memory corruption, which can potentially lead to security vulnerabilities."1

Rust's fix is radical and simple. After let s2 = s1;, it copies the three stack fields (pointer, length, capacity) — but not the heap data — and then declares s1 invalid. Only s2 owns the text now (rule 2: one owner). So only one drop ever runs.

★ The key word: move

"Because Rust also invalidates the first variable, instead of being called a shallow copy, it's known as a move. In this example, we would say that s1 was moved into s2."1

If you know C++: this is the default behaviour of std::move / unique_ptr — ownership transfers — but applied automatically to every heap-owning type, and enforced by the compiler so the moved-from variable can't be touched.

So the snippet above does not compile. Rust stops you at build time with a famously helpful error:

error[E0382]: borrow of moved value: `s1`
 --> src/main.rs:5:15
  |
3 |     let s2 = s1;
  |              -- value moved here
4 |
5 |     println!("{s1}, world!");
  |               ^^^^ value borrowed here after move

Note the design choice this encodes: Rust never deep-copies your data automatically. Any automatic operation is cheap (just a few stack words).1 If you want a real deep copy, you ask for it by name with .clone() — more on that in the exercises.

5The exception: Copy types

But you've surely written let y = x; with integers a thousand times and used both afterwards. Why doesn't that move? Because integers live entirely on the stack — there's no heap allocation to have two owners of, so copying them is trivially cheap and safe.

Rust marks such types with the Copy trait: assignment copies the value and both variables stay valid.1 The rule of thumb:

BehaviourWhich typesAfter let b = a;
Copy (stays valid)Stack-only scalars: all integers, f64, bool, char, and tuples of only those (e.g. (i32, i32)).1a still usable ✓
Move (invalidated)Anything owning a heap allocation or resource: String, Vec<T>, Box<T>, file handles… (i32, String) too.a is dead ✗

One clean line from the book to keep: anything "that requires allocation or is some form of resource" cannot be Copy.1

6Practice — predict, then verify

This is the loop. For each snippet: decide in your head whether it compiles (and if not, which variable is the problem), then open the reveal, then hit ▶ Run to confirm with the real compiler. Reason from the rules, not from memory of other languages.

Exercise 1 — does this compile? 
fn main() {
    let a = String::from("rust");
    let b = a;
    println!("{a} and {b}");
}
Reveal

No. String is heap-owning, so let b = a; moves ownership to b and invalidates a (rule 2). The {a} then fails with error[E0382]: borrow of moved value: 'a'. b alone would be fine.

Read the error. The "value moved here" / "value borrowed here after move" annotations point at the exact two lines.

Exercise 2 — and this one? 
fn main() {
    let x = 5;
    let y = x;
    println!("{x} and {y}");
}
Reveal

Yes. 5 is an i32 — a stack-only Copy type. let y = x; copies it; x stays valid. Same syntax as Exercise 1, opposite outcome — and the only thing that changed is whether the type owns heap memory. That distinction is the whole game.

Prints 5 and 5.

Exercise 3 — fix Exercise 1

You want two independent strings. .clone() makes a real deep copy of the heap data, so each variable owns its own. Predict: does this compile, and what prints?

fn main() {
    let a = String::from("rust");
    let b = a.clone();
    println!("{a} and {b}");
}
Reveal

Yes — prints rust and rust. a.clone() allocates a second heap buffer and copies the bytes, so a is never moved; both own separate data and both get dropped exactly once. The cost (an allocation + copy) is now visible in the code — that's the design: cheap things happen silently, expensive things you spell out.

Exercise 4 — the cliffhanger

Passing a value to a function is an assignment too — so it moves. Predict carefully:

fn main() {
    let s = String::from("hello");
    shout(s);
    println!("{s}");
}

fn shout(text: String) {
    println!("{}!", text);
}
Reveal

No. Calling shout(s) moves s into the parameter text; ownership leaves main. At the end of shout, text is dropped and the string is freed. Back in main, s is invalid → E0382 again.

This feels too strict, right? You just wanted to look at the string, not hand it over forever. That tension is exactly what the next lesson resolves — you'll borrow a value with & instead of moving it. Sit with the discomfort; it's the setup.

7Why this matters for calling Rust from Python

⌖ Mission link

Your goal is a Rust core that Python can import. When Rust hands a value across that boundary — or Python hands one in — something has to decide who owns it and who frees it. Get that wrong in a C extension and you get exactly the double-frees and use-after-frees from §1, now crashing the Python interpreter.

Ownership is what makes the Rust↔Python boundary safe by construction: the binding layer (PyO3) is built on these exact move/borrow rules. It's also why Rust can safely release Python's GIL and run real parallel work — because the compiler has already proven nothing is sharing what it shouldn't. Everything you just learned is load-bearing for the milestone.
