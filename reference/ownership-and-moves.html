Rust → Python · Quick ReferenceOwnership & Moves

Who owns it → who frees it. Enforced at compile time.

The three rules

  1. Each value has one owner.
  2. There can be only one owner at a time.
  3. When the owner leaves scope, the value is dropped (freed) — automatically, via drop. You never write free.

Move vs Copy vs Clone

WhatHeap dataSource after
MOVE let b = a;shared (ptr copied)invalid
COPY let b = a;none (stack only)still valid
CLONE let b = a.clone();duplicated (new alloc)still valid

Auto operations are always cheap. A deep copy never happens unless you write .clone().

Move or Copy? (by type)

COPYStack scalars: integers, f64, bool, char, and tuples of only those — e.g. (i32, i32).
MOVEOwns heap/resource: String, Vec<T>, Box<T>, files… and any tuple/struct containing one, e.g. (i32, String).

Rule: "anything that requires allocation or is a resource" cannot be Copy.

Will a still work after let b = a;?

Is a's type stack-only (Copy)?  → YES: copied, both valid.  |  NO: moved, a is dead → error[E0382].
Need both and it's a heap type?  → use a.clone() (pay for a real copy, on purpose).

The classic C/C++ bug → Rust's answer

C/C++ bugRust prevents via
Forget free (leak)Rule 3: auto drop at scope end
Use-after-free / dangling ptrMoved-from value is invalid → won't compile
Double-freeRule 2: one owner → exactly one drop

Same idea as C++ RAII / unique_ptr — but the language-wide default, compiler-enforced, with zero runtime cost.

Snippets

// MOVE — won't compile
let a = String::from("hi");
let b = a;          // a moved into b
println!("{a}");    // E0382: a is dead

// COPY — fine
let x = 5;
let y = x;          // both valid

// CLONE — fine, explicit cost
let a = String::from("hi");
let b = a.clone();  // two owners, two buffers

// Functions move too
fn shout(t: String) { /* t owned, dropped here */ }
let s = String::from("hi");
shout(s);
println!("{s}");    // E0382 — fixed by borrowing (next)
